## 2. 原 SO 架构（已核实）

```
libJCC.so
  ├─ 字符串 resolve 类/方法（BuyHeroView, OnRefreshHeroRet, ReqBuyHero,
  │    SearchACGHero*, GetMatchPlayerId, PlayerModel, HextechAugmentsCtrl…）
  ├─ DobbyHook 安装多组钩子
  ├─ 字段：LDR/STR 立即数（无 il2cpp_field_from_name）
  └─ TCP 31338 对 UI 输出 GET/SET/DO/PUSH
```

## 3. 关键证据：很多「游戏对象字段」其实仍匹配本赛季

### 3.1 英雄表解包 `0x7e4bc`（完整 MATCH）

| SO 读的偏移 | 扫描 TACG_Hero_Client |
|-------------|------------------------|
| 0x10 | iID |
| 0x18 | sName |
| 0x34 | iStar |
| 0x38 | iQuality |
| 0x60 | iCost |
| 0xf0 / 0xf8 | sHeroPaint / Small |

→ **静态牌库读表不是闪退主因。**

### 3.2 对手/玩家信息链 `0x84a90` 一带（也大量 MATCH）

在 resolve/调用之后对对象读取：

| SO 偏移 | 扫描 PlayerModel |
|---------|------------------|
| 0x20 | battleTurnModel |
| 0x5c | m_money |
| 0x64 | **LastEnemyId（上一/相关敌人 ID，和「对手」强相关）** |
| 0xbc | _iPlayerHpLeft |
| 0x268 | freeRefreshCnt（同偏移也在 ChessBattleModel.my_match_list） |

→ **「下一局对手」相关的若干 PlayerModel 字段，本赛季仍对得上。**  
→ 因此不能简单说「对手功能整段偏移全废」。

### 3.3 全 .text 对象 LDR 统计

在扫描的 17 个关键类字段集合里：

- 绝大多数 SO 使用的对象字段立即数 **都能在扫描表中找到同值**
- 明显「扫描表完全没有」的只有少量（如 **0x3c** 出现约 20 次，**0xa2** 约 12 次）—— 可能是未扫到的类或 SO 自有结构

### 3.4 易误判：SO 自己的 C++ 懒加载槽

例如 `0x8cad4` 一带：

```text
ldrb [x19,#0x258]  ; 标志位
ldr  [x19,#0x268]  ; 指针
ldrb [x19,#0x240]
ldr  [x19,#0x250]
...
```

这是 **SO 内部对象** 上「bool + ptr」成对布局，**不是** ChessBattleModel 的 match 列表。  
之前把大量 `#0x250/#0x268` 当成「游戏字段过期」会误判。

## 4. 那闪退更可能出在哪（分析后优先级）

在「很多字段仍 MATCH」的前提下，闪退更可能是：

| 优先级 | 原因 | 说明 |
|--------|------|------|
| **A** | Hook 装上了但目标方法无效 / 回调里假设不成立 | `OnRefreshHeroRet` 在 dump 里几乎只有 IDMAP，方法体可能空或签名变；一点商店相关就崩 |
| **B** | 中间指针链某环类型变了 | 如 `[obj+0x190]` 再 `+0x10`；字段还在但语义/类型变了 → 解引用崩 |
| **C** | 未扫全的类上的真正错偏移 | 如 0x3c 等不在当前 17 类表里 |
| **D** | 注入时机/多线程 | 较少表现为「点某个功能才崩」 |

**不是：**「英雄 0x10/0x18/0x60 全错了」（已否定）。

## 5. 方法符号（dump 对照）

| 符号 | 原 SO | 当前 dump |
|------|-------|-----------|
| OnRefreshHeroRet | 有 | 几乎仅 IDMAP |
| ReqBuyHero | 有 | 有（HeroRoot 等） |
| GetMatchPlayerId | 有 | 有 |
| SearchACGHero2 | 有 | 有 |
| HextechAugmentsCtrl | 有 | 有 |
| HandleRefreshBuyHero | **无** | 有（ChessBattleStage） |
| ChessBattleUnit / UnitData 字符串 | **无** | 类存在 |

若 `OnRefreshHeroRet` 运行时 resolve 失败或 hook 到坏地址 → **一点相关功能就闪退**，这比「英雄偏移错」更符合「点了才崩」。

## 6. 工程含义（分析清楚后的做法）

### 已能确定

1. 方向：补原 SO 的读点，不重写业务。  
2. 英雄表字段：**无需 patch**。  
3. PlayerModel 金钱/血量/LastEnemyId：**当前指令与扫描一致**。  
4. 不能对「所有 unknown 立即数」盲改（含 SO 内部结构）。

### 下一步才算「开始修闪退」（有证据再改指令）

1. **扩大扫描类表**（覆盖 SO 用到但未扫的类），确认 0x3c 等归属。  
2. **运行时原版 log**（`files/log.txt`）：哪组 Hook 是 0/N、FATAL 行——**这是区分 A/B/C 的最短路径**（不是让用户猜，是读程序自己打的日志）。  
3. 若 log 显示 `OnRefreshHeroRet` 失败 → **改绑到现赛季仍存在的刷新回调**（代码级，不是瞎改偏移）。  
4. 若 log 显示 hook 全成功仍崩 → 再对崩溃点附近 **唯一不匹配的对象字段** 做 ARM64 重编码 patch。

## 7. 一句话

**闪退确实是「读内存的数据/布局问题」；但细查后，英雄和多个 PlayerModel 关键字段并没过期。更可疑的是商店刷新 hook 目标失效，或中间对象链/未扫类字段。下一步必须用原版 log + 扩大扫描钉死那一环，再改 SO，而不是整表盲改偏移。**

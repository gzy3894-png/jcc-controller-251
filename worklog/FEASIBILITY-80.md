## 为什么不是 100%，也不是 50%

### 推高置信的证据（硬）

1. **英雄解包函数 `0x7e4bc`**  
   `0x10/0x18/0x34/0x38/0x60/0xf0/0xf8` 与扫描 `TACG_Hero_Client` **全部一致**。  
   → 静态牌库读数路径 **不需要改偏移**。

2. **对手相关使用点 `0x84aa8` 一带**  
   对对象读取：  
   - `0x5c` = `PlayerModel.m_money`  
   - `0xbc` = `_iPlayerHpLeft`  
   - `0x64` = **`LastEnemyId`**（与「下一局/对手」直接相关）  
   - `0x20` = `battleTurnModel`  
   与扫描 **一致**。  
   → 「下一局对手」**不是整段偏移全废**，而是 **有可用读点**。

3. **原 SO 符号仍在**  
   `ReqBuyHero`、`GetMatchPlayerId`、`SearchACGHero*`、`HextechAugmentsCtrl`、`PlayerListPanel`、`UpdateBattleMap`、`DataBaseManager` 等均在 SO 内，且 dump 的 **IDMAP 仍登记**（说明游戏元数据里方法仍在；Zygisk dump 几乎不吐方法体，不能据此说方法消失）。

4. **已定位真·赛季断裂点（可修）**  
   - 原 hook：`BuyHeroView.OnRefreshHeroRet`  
   - 当前 `BuyHeroView` 方法列表（完整扫类）**没有** `OnRefreshHeroRet`（仅 IDMAP 残留）  
   - 替代：`ChessBattleStage.HandleRefreshBuyHero(Object)` **存在**，argc=1 与原 resolve 的 `w3=#1` 一致  
   - **2.5.4 已做该改绑补丁**（指令 + 新字符串）

5. **闪退机理与「数据位置」一致**  
   hook 到不存在的方法 / 空回调 → 一点相关功能就崩，符合用户描述。

### 拉低置信的点（故压在 82% 而非 95%）

1. Zygisk dump **几乎没有方法体**，多数方法只能靠 IDMAP 判断「还在」，不能看签名细节。  
2. 部分路径中间指针（如 `+0x190` 再解引用）**类型是否变化**未在运行时验证。  
3. `PlayerHeadInfo` 在定向扫描中 hits=0，列表/头像可能有额外改名。  
4. 自动拿牌还依赖商店槽位对象布局，**未做槽位级逐字段证明**。  
5. 曾把 SO **内部**懒加载槽（`#0x250/#0x268` 成对）误判成游戏字段——说明必须「使用点级」对照，不能全表盲改。

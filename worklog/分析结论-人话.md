# 分析结论（人话版）

## 1. 你的现象怎么解释

**一点功能就闪退** = 程序按「旧地图」去内存里取值，取到非法地址 → 崩。  
这就是 **数据/结构布局变了**，不是你要猜、也不是功能代码丢了。

原版 SO 里 **功能怎么实现已经写好了**（含内存自动拿牌、读下一局对手等）。  
坏的是它 **读的位置** 有一部分过期了。

---

## 2. 原 SO 怎么工作（已搞清）

```
注入 → 找到 libil2cpp
     → 用「类名/方法名字符串」找到游戏函数
     → Dobby 挂钩（商店刷新、买牌、地图更新…）
     → 用指令里的固定偏移（LDR #0x??）读对象字段
     → 通过 31338 把结果给界面
```

所以要修的是：

1. **方法还能不能找到**（名字/签名变了会 hook 失败，后面一用就崩或空）  
2. **字段偏移对不对**（错了就闪退）  

不是重写「自动买算法」「预测算法」。

---

## 3. 对照新赛季扫描后的结果

### 3.1 英雄表（静态牌库）—— **没问题**

原 SO 里解英雄的函数（地址 `0x7e4bc`）读的是：

| 偏移 | 含义 | 新赛季扫描 |
|------|------|------------|
| 0x10 | id | 一致 |
| 0x18 | 名字 | 一致 |
| 0x34 | 星级 | 一致 |
| 0x38 | 品质 | 一致 |
| 0x60 | 费用 | 一致 |
| 0xf0 / 0xf8 | 大/小头像 | 一致 |

→ **这块不是闪退主因。**  
→ 所以你会感觉「商店/牌库信息有时还能动」，和这个一致。

### 3.2 按功能看（字符串定位 + 附近字段读）

| 功能 | 原 SO 符号 | 分析结论 |
|------|------------|----------|
| 商店刷新 | `OnRefreshHeroRet` | 能定位；附近立即数都能在扫描表找到同值（仍要确认类） |
| **自动拿牌** | `ReqBuyHero` | 能定位；**有 3 个立即数对不上扫描表 → 优先怀疑** |
| **下一局对手** | `GetMatchPlayerId` | 能定位；**约 20 个立即数对不上 → 闪退重灾区嫌疑** |
| 自己 | `GetMyPlayerModel` | 附近较干净 |
| 头像列表 | `PlayerListPanel/Item` | **多处对不上** |
| **海克斯** | `HextechAugmentsCtrl` | 能定位；有未对齐立即数 |
| 站位推送 | `OPPONENT_BOARD` | **多处对不上** |
| 对战模型 | `ChessBattleModel` | **unk 很多** |
| 战斗单位 UnitData | 字符串不在 SO 里 | 走别的类名/间接访问，更难点名 |

说明：

- 「对上」= 这个数字在新赛季某个类的字段表里出现过（还要确认是不是同一个类）。  
- 「对不上」= **更像旧布局残留 → 一点击就容易崩的地方**。

## 4. 总判断（清楚、不绕）

| 判断 | 内容 |
|------|------|
| 闪退原因 | **数据/布局过期导致错误读内存**（你说的对） |
| 功能代码 | **还在原 SO 里**，不用重写业务 |
| 英雄表偏移 | **这季不用改** |
| 闪退主战场 | **下一局对手、站位/列表、对战模型、自动买相关读点** 等「对不上」的立即数 |
| 现在能不能说全好了 | **不能**；分析清楚了主因和优先修哪里，**补丁还没做完** |
